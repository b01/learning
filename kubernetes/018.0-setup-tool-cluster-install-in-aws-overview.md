# Setup Tool Cluster Install in AWS Overview

TLDR; This guide teaches you how to prepare to set up a `self-hosted` Kubernetes
cluster, then install the `AWS Load Balancer Controller`. It's ONLY tested for
clusters running on EC2 instances in AWS Cloud, and without EKS.

If you feel you messed up, you can use kubeadm to reset (delete) the cluster
and start over, see [You Should Know kubeadm Reset]

## Prerequisites

### Fix Internet Access for IPv4

WARNING: This method will cost you money. If you forget to delete
the NAT gateway for days, it can add up quickly. That said, also you need to
remember to shut down the EC2 instances when you're not using them, or you plan
to break for more than 30 minutes; as they rackup cost every minute.
So the NAT Gateway you delete when not using, and the EC2 instance you just need
to shut-down.

If you don't want to use S3 as a proxy, you can deploy a NAT gateway. Then add
rules to routing tables attached to private subnets for kubernetes. This will
fix IPv4 internet access for your EC2 instances in the Dual stack VPC.

1. Deploy a NAT to the VPC containing your cluster.
2. Update the route table of all private subnets in the VPC where you deployed
   your cluster nodes:
    1. Add a route for IPv4 traffic to the NAT `0.0.0.0/0 -> NAT`.
    2. Add a rote for IPv6 out to the NAT `64:ff9b::/96 -> NAT`.

NOTE: `64:ff9b::/96` is the IETF-reserved "Well-Known Prefix" (WKP) used for
NAT64 translation, allowing IPv6-only clients to communicate with IPv4-only
servers. It enables embedding a 32-bit IPv4 address into the last 32 bits of an
IPv6 address, commonly used in DNS64/NAT64 environments.

### Add an S3 Bucket Proxy

This bucket will be used to store files so that you can deploy them to multiple
nodes quickly.

1. Make a new S3 bucket in the same account as your cluster. You can do this
   through the console or CLI.
2. Add an IAM Policy to the IAM role used by your Kubernetes nodes to allow
   access to that S3 bucket.
3. Log into each EC2 instance and install the AWS CLI.
4. In order to download from S3 over IPv6, you'll change the AWS CLI config for
   S3 to allow it to work over IPv6:
   ```shell
   mkdir ~/.aws
   cat <<TXT | tee -a ${HOME}/.aws/config
   [default]
   s3 =
       use_dualstack_endpoint = true
       addressing_style = virtual
   TXT
   ```
5. Back on your local machine, download the following files to a new directory,
   call it `kubernetes-assets`:
    1. https://github.com/cert-manager/cert-manager/releases/download/v1.20.0/cert-manager.yaml

   NOTE: Use any method you like to download. Also, it does not matter what you
   call the directory, we chose a name to make it simple to write this guide.
